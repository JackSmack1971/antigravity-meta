The Agent Skill Architecture: A Foundation for ExcellenceA deep understanding of the underlying Agent Skill architecture is not merely academic; it is the strategic foundation for designing efficient and scalable Skills. This knowledge empowers developers to design for performance, control token usage, and unlock capabilities that are impossible to achieve with prompts alone. Mastering these architectural principles is the key to equipping Claude with vast amounts of specialized knowledge without overwhelming its context window.An Agent Skill is a reusable, filesystem-based resource that packages instructions, metadata, and optional code. Think of it as an onboarding guide for a new hire. In contrast to single-use, conversation-level prompts, the Skill model provides specialization, reusability, and the ability to compose multiple Skills to build complex, sophisticated workflows.The central design principle behind this architecture is Progressive Disclosure. This powerful concept ensures that Claude loads information in stages as needed, rather than consuming the entire context of a Skill upfront. There are three distinct levels of loading content, a tiered approach that enables Claude to be aware of a vast library of Skills while only loading the precise information required for a given task.Three Levels of Skill Content LoadingLevel When Loaded Token Cost & ContentLevel 1: Metadata Always (at startup) ~100 tokens per Skill. Includes the name and description from the SKILL.md YAML frontmatter. This allows Claude to know the Skill exists and when to use it.Level 2: Instructions When the Skill is triggered Under 5k tokens. The main body of the SKILL.md fileâ€”the new hire's desk referenceâ€”is read from the filesystem into the context window.Level 3+: Resources As needed Effectively unlimited. Bundled files (additional .md files, scripts, data) are accessed or executed via bash. The contents of executable scripts are never loaded into context; only their output is.This filesystem-based architecture confers several strategic advantages that enable highly efficient and powerful Skill design.* On-demand file access: Claude reads only the files needed for a specific task. A Skill can contain dozens of reference files, but only the one relevant to the user's request will be loaded into context, consuming zero tokens for the rest.* Efficient script execution: When Claude runs a bundled script, the code itself never enters the context window. Only the scriptâ€™s output (e.g., "Validation passed" or an error message) consumes tokens, making scripts far more efficient and reliable for deterministic operations than generating code on the fly.* No practical limit on bundled content: Because files do not consume context until they are accessed, Skills can include comprehensive API documentation, large datasets, or extensive examples without incurring an upfront token penalty.Understanding this architectural "how" is the first step. The next is to apply this knowledge to the principles of "what" to write within your Skills.--------------------------------------------------------------------------------2. Information Architecture: Designing for Progressive DisclosureThe information architecture of a Skill is how you organize the "company knowledge base" for your new "employee," Claude. The goal is to fully leverage the progressive disclosure model, structuring your instructions, references, and code in a way that minimizes token usage while maximizing clarity. A well-designed architecture ensures that Claude can find exactly what it needs, precisely when it needs it.There are three primary patterns for structuring Skill content effectively.1. High-Level Guide with References: In this common pattern, the SKILL.md file acts as a "table of contents" or a quick-start guide. It provides the most essential information upfront and then points to more detailed files for advanced features, specific workflows, or comprehensive API references. This keeps the initial context load lean and focused.2. Domain-Specific Organization: For Skills that cover multiple distinct domains (e.g., a data analysis Skill that can query sales, finance, and product databases), partition the reference material by subject matter. Create separate files like reference/finance.md and reference/sales.md. This isolates context, ensuring that a query about sales metrics only loads sales-related schemas, not irrelevant financial data.3. Conditional Details: Structure the Skill to present basic information or common workflows directly in SKILL.md while linking to separate files that contain instructions for more advanced, conditional, or specialized tasks. For example, a document editing Skill might cover standard creation in the main file but link to REDLINING.md for instructions on handling tracked changes.Structural Best Practices* Avoid Deeply Nested References: All reference files should be linked directly from the main SKILL.md file. Nesting references (e.g., SKILL.md links to advanced.md, which in turn links to details.md) can cause Claude to only partially read the final file, leading to incomplete context. Keep your information architecture one level deep for maximum reliability.* Structure Longer Reference Files with a Table of Contents: For any reference file exceeding 100 lines, include a table of contents at the top. This allows Claude to understand the full scope of the file's content even if it initially performs a partial read, enabling it to navigate to the correct section more efficiently.With a well-defined architecture in place, you are ready to implement more advanced patterns for guiding Claude through complex workflows.--------------------------------------------------------------------------------5. Best Practices for Skills with Executable CodeWhen a Skill bundles executable scripts, a fundamental shift occurs. You are no longer just guiding a probabilistic reasoner (Claude); you are commanding a deterministic system (your code). Executable code is the ultimate "escape hatch" for reliability, used when the ambiguity of natural language is too high a risk for a given task. This power demands a more rigorous architectural approach.* Solve, Don't Punt: Your scripts must be built with robust error handling to manage foreseeable issues directly. Do not simply fail and "punt" the problem back to Claude. A script that gracefully handles a FileNotFoundError by creating a default file is an order of magnitude more useful than one that crashes.* Good example: Handles errors explicitly* Bad example: Punts error to Claude* Provide Verifiable Intermediate Outputs: For high-stakes or destructive operations, use a "plan-validate-execute" pattern. Have Claude first generate a plan (e.g., a JSON file of changes to be made), then use a validation script to check that plan for errors. Only after the plan is verified should Claude execute the final step. This catches errors before any irreversible changes are made.* Manage Dependencies & Environment Constraints: The runtime environment for Skills has critical differences depending on the surface. The claude.ai environment offers flexibility for rapid prototyping, allowing installation of packages from PyPI, npm, and GitHub. In contrast, the Claude API environment is a locked container with no network access, requiring a more rigorous, production-oriented approach where all dependencies must be pre-baked. Always check the official code execution tool documentation for a list of available packages.* Reference MCP Tools Correctly: If your Skill uses Model Context Protocol (MCP) tools, you must use the fully qualified tool name in the format ServerName:tool_name (e.g., BigQuery:bigquery_schema). Failing to include the server prefix can result in "tool not found" errors, especially when multiple MCP servers are available.From the technical details of creation, we now shift our focus to the crucial lifecycle of testing and improving a Skill over time.--------------------------------------------------------------------------------7. The Iterative Development Lifecycle: Evaluation and RefinementSkill authoring is a dynamic, iterative lifecycle, not a one-time task. The most effective Skills are not just written; they are continuously refined through a structured process of testing, observation, and improvement based on real-world usage.Adopt Evaluation-Driven DevelopmentInstead of anticipating requirements, solve for observed needs. This approach ensures your Skill addresses real problems and provides measurable value.1. Identify gaps: Run Claude on representative tasks without a Skill and document where it fails or lacks context.2. Create evaluations: Build a few test scenarios that specifically target these identified gaps.3. Establish a baseline: Measure Claudeâ€™s performance on your evaluations without the Skill.4. Write minimal instructions: Author just enough content in your Skill to address the gaps and pass the evaluations.5. Iterate: Execute the evaluations, compare the results against the baseline, and refine the Skill based on performance.Develop Skills Iteratively with ClaudeThis powerful meta-technique uses Claude as a development partner. The workflow involves using one Claude instance ("Claude A") to help author and refine the Skill, while a separate, fresh instance ("Claude B") is used for testing. After working through a task with Claude A, ask it to "create a Skill that captures this pattern we just used." Claude understands the Skill format natively and can generate a well-structured starting point.To accelerate this process, you can use the built-in "skill-creator" skill on Claude.ai. This meta-skill provides interactive guidance, asking about your workflow and generating the necessary folder structure and SKILL.md file, significantly streamlining the initial setup.Observe How Claude Navigates SkillsTesting is more than just pass/fail; it is an opportunity to observe Claude's behavior and gain insights for refinement. During testing, watch closely for the following:* Unexpected exploration paths: Does Claude read files in an order you didn't anticipate? This may signal that your information architecture could be more intuitive.* Missed connections: Does Claude fail to follow a critical reference to another file? The link in your SKILL.md may need to be more prominent or explicit.* Ignored content: If Claude consistently ignores a bundled file or section, it may be poorly signaled, or it may not be necessary at all.These direct observations of Claude's interaction with your Skill are the primary drivers for meaningful refinement and improvement.The development lifecycle ensures your Skill is robust and effective. Security and Deployment ConsiderationsIntegrating custom Skills into organizational workflows requires a sharp focus on security and a clear understanding of the deployment landscape. This final step is crucial for ensuring that the power of Skills is harnessed safely and that they are available on the correct platforms for your users.Prioritizing SecurityA Skill gives Claude new capabilities through instructions and executable code. While this is what makes them powerful, it also introduces potential security risks. A malicious Skill could direct Claude to exfiltrate data, access unauthorized systems, or perform other harmful actions.* Audit thoroughly: Before using any Skill from an external source, review all its bundled files, including scripts, markdown, and other resources.
